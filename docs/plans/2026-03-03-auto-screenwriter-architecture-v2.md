# Auto-Screenwriter Architecture V2 (架构设计 V2 版)

基于我们关于字节跳动 DeerFlow 和 Milvus 的深入讨论，我们将 Auto-Screenwriter 的底层核心架构正式升级为 V2 规范。这份规范将指导未来所有的代码开发。

## 1. 核心多智能体拓扑 (Supervisor Pattern 调度模式)

在这个版本中，我们不再是简单的线性流水线，而是采用 DeerFlow 倡导的**“主管-工人”隔离模式**。

*   **`planner_node` (总导演 & 主管)**: 它**不负责**具体的文字生成。它的职责是接收用户的长需求，制定剧本大纲，维护“世界观状态”，并在未来负责派发并行任务给多个 Writer 子智能体。
*   **`writer_node` (打工编剧)**: 一个极其纯粹的生成节点。它的“脑容量”里只有当前需要写的几行对话或者一组分镜动作。它会在需要时，主动按需拔出工具（如`search_story_bible`）去找资料。
*   **`critic_node` (品控审核)**: 它是出厂前的最后一道关卡。负责对输出的格式（比如好莱坞编剧格式是否正确）和情节是否出现逻辑冲突进行自检，如果不通过，打回去让 Writer 重写。
*   **`reflector_node` (记忆清扫员 - Plan)**: 未来将加入的一个异步节点，负责在剧本写完后，把 3000 字正文压缩成 50 字的摘要，存入账本。

## 2. 存储与记忆的三级隔离引擎 (The V2 Data Layer)

这是对原有简陋 RAG 架构的最大颠覆，我们将严格区分“流动记忆”和“静态素材库”：

*   **L0 (内存结构体 - LangGraph State)**:
    *   **载体**: Python `typing.TypedDict` (结合 Checkpointer)。
    *   **作用**: 这是**流动的血液**。里面只存放当前回合用户的 prompt、当前的报错信息、由于修改引发的轮次（Revision Count）。它会高频刷新，阅后即焚，坚决防止大模型的上下文爆炸。
*   **L1 & L3 (事务型状态账本 - Acid Ledger)**:
    *   **载体**: 未来接入 **PostgreSQL** (通过 `JSONB` 字段存储半结构化数据)。
    *   **作用**: L1 存放项目的一句话总结、用户设置的编剧偏好（比如“不许出现暴力场景”）；L3 扮演 `World Tracker`（世界追踪器），类似于游戏里的血条面板，精确记录当前写到了哪一集的哪场戏，哪个道具在谁手上。这是**防止模型吃书、胡编乱造**的最强硬核锁。
*   **L2 (海量情境检索库 - The RAG Vectors)**:
    *   **载体**: **Milvus** (MVP 阶段采用无需 Docker 的 `Milvus Lite` 本地嵌入版)。
    *   **作用**: 被贬为纯粹的“静态被动维基百科”。存放几百万字的原著小说、人物历史档案、或者过去已经定稿的剧本。
    *   **调用方式**: **只有**当 `writer_node` 卡壳，需要查“李四以前用过什么剑法”时，才会主动拉起工具调用 `Milvus`。查完立刻结束，绝不干扰主流程。

## 3. 动态技能挂载 (Progressive Skill Context)

我们将全面吸收 DeerFlow 的 `.md` 文件规范。我们将建立一个 `app/skills/` 文件夹。
*   如果今天 Planner 让 Writer 去写环境描写，系统就把 `world-building-rules.md` 塞给它。
*   如果要写人物对话，就换成 `dialogue-formatting.md`。
*   大模型永远保持最高专注度，绝不会因为“需要遵守100条系统设定”而导致 AI 智商下降。
